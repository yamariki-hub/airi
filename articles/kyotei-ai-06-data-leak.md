## 「ROI 148%」という数字の正体

バックテストが終わった翌朝、もう一度コードを読み返した。なんとなく「ありえない数字だ」という違和感があったからだ。

ヘッジファンドにいたとき、先輩クォンツに言われた言葉がある。

「バックテスト結果が良すぎると感じたら、必ずリークを疑え。100%を大きく超えるリターンは、現実ではなくデータの問題だ。」

コードを3時間かけて読み返した。そして見つけた。

```python
# v1の特徴量計算コード（NG例）
df["racer_win_rate_30d"] = (
    df.groupby("racer_number")["place"]
    .transform(
        # 問題: shift(1)がない
        # これにより「当日の結果」が「当日の予測特徴量」に含まれる
        lambda s: (s == 1).rolling(30, min_periods=5).mean()
    )
)
```

**たった `shift(1)` の欠落。** これだけで全てが狂った。

## データリークとは何か：「答えを見ながら解く」状態

データリークを理解するには、試験を例にするとわかりやすい。

想像してほしい。「明日のテストで何が出るか予測しろ」という課題がある。正しいアプローチは「今日まで習ったことから推測する」ことだ。

でも、もし「明日のテスト問題を見てから予測を作った」としたら？当然、予測精度は100%に近くなる。でもそれは実際の予測能力ではない。

競艇AIの文脈で言うと：

- **本来あるべき姿**: 「今日のレース前に得られる情報」から「今日の着順」を予測
- **データリーク状態**: 「今日の着順（結果）」が「今日の特徴量」に混入している

`shift(1)` なしの勝率計算では、「今日A選手が1着になった」という情報が、「今日のA選手の勝率特徴量」に含まれてしまう。バックテストで評価するとき、モデルは「答えを見ながら問題を解く」状態になっている。

```python
# NG: 当日の結果が含まれる
lambda s: (s == 1).rolling(30).mean()

# OK: shift(1)で前日以前の情報のみ使用
lambda s: (s.shift(1) == 1).rolling(30, min_periods=5).mean()
```

この1行の差が、ROI 148%と91%の差だ。

## v1モデルの問題点：3つの致命的な欠陥

振り返ると、v1には3つの根本的な問題があった。

### 問題1: データリーク（最も深刻）

上で詳しく説明した通り。全ての「勝率」「2連率」「平均ST」の計算に `shift(1)` が抜けていた。

特に危険だったのは、「計算は動く」「結果も出る」「エラーも出ない」という点だ。コードは正常に実行される。ただし「答えを見ながら解く」状態になっているだけで。

データリークはサイレントキラーだ。

### 問題2: race_cardsテーブルが空だった

当初、出走表データ（選手成績・モーター成績）を全く収集していなかった。

`race_cards` テーブルは存在するが、レコード数ゼロ。モーター2連率も、選手の級別も、全国勝率も、展示タイムも——全部欠落。

でも**LightGBMが欠損値を自動処理してくれるので、エラーにならない。** `NaN` を受け取ったLightGBMは「この特徴量がない場合はこう処理する」という分岐を学習する。パフォーマンスが落ちているのに、コードが動くから気づかない。

これを発見したのは、こんなコードを書いたときだ：

```python
from sqlalchemy.orm import Session
from db import engine, Race, RaceCard

def check_data_coverage():
    """データのカバレッジを確認する"""
    with Session(engine) as session:
        # レース結果の件数
        total_races = session.query(Race).count()
        print(f"レース数: {total_races}件")

        # 出走表データの件数
        card_count = session.query(RaceCard).count()
        print(f"出走表データ: {card_count}件")

        # カバレッジ（出走表があるレースの割合）
        if card_count > 0:
            from sqlalchemy import func
            races_with_cards = (
                session.query(func.count(func.distinct(RaceCard.race_id)))
                .scalar()
            )
            coverage = races_with_cards / total_races * 100
            print(f"カバレッジ: {races_with_cards}/{total_races} ({coverage:.1f}%)")
        else:
            print("出走表データ: ゼロ件（収集できていません）")
```

実行結果：
```
レース数: 34,121件
出走表データ: 0件
出走表データ: ゼロ件（収集できていません）
```

ゼロ。

最重要特徴量の全てが欠落した状態で8ヶ月間モデルを動かしていた。LightGBMが欠損値をうまく処理するからこそ気づかなかった。

この発見から出走表スクレイピングを実装し、22,000レース分を取得した。それが約4時間の作業だった。

### 問題3: 穴狙いの思い込み

AIを作る前から「AIなら人気薄（穴）を当てられる」という先入観があった。

「市場は過大評価と過小評価を繰り返す」「群衆の判断は間違っている」——ヘッジファンドで培ったそういう感覚が、競艇にも通じると思っていた。

実際のデータを見ると：

- **1号艇**: 勝率 54.6%　平均配当 166円　均等ベットROI **90.4%**
- **2号艇**: 勝率 13.2%　平均配当 513円　均等ベットROI 67.8%
- **3号艇**: 勝率 12.8%　平均配当 579円　均等ベットROI 74.4%
- **4号艇**: 勝率 10.2%　平均配当 693円　均等ベットROI 70.7%
- **5号艇**: 勝率 6.1%　平均配当 1,036円　均等ベットROI 63.2%
- **6号艇**: 勝率 3.1%　平均配当 1,172円　均等ベットROI **36.2%**

**穴（6号艇）が最も損する。**

6号艇の平均配当は1,172円と高い。でも勝率は3.1%しかない。100回賭けると平均3回しか当たらず、36.2円しか戻ってこない計算だ（100円ベースで）。

競艇市場は穴を正確に嫌っている。「難しい艇ほど配当が高い」のは当然だが、**それ以上に勝率が低い**のだ。市場は正しい。少なくとも「穴は過小評価されている」という私の仮説は完全に否定された。

## データリーク検出の方法：自分のコードを疑え

自分のコードにリークが入っていないか確認する方法を2つ紹介する。

### 方法1: シャッフルテスト（最も確実）

```python
import lightgbm as lgb
import numpy as np

def shuffle_test(X: np.ndarray, y: np.ndarray, params: dict) -> tuple[float, float]:
    """
    シャッフルテスト: ターゲットをランダムにシャッフルして学習・評価する。
    リークがない場合: 通常ROI >> シャッフルROI
    リークがある場合: 通常ROI ≈ シャッフルROI（シャッフルしても精度が高い = 答えが特徴量に入っている）
    """
    # 通常の学習・評価
    model_normal = lgb.train(params, lgb.Dataset(X, y), num_boost_round=200)
    roi_normal = evaluate_roi(model_normal, X, y)

    # ターゲットをランダムにシャッフルして学習・評価
    y_shuffled = y.copy()
    np.random.shuffle(y_shuffled)  # ラベルをシャッフル（ランダム化）
    model_shuffled = lgb.train(params, lgb.Dataset(X, y_shuffled), num_boost_round=200)
    roi_shuffled = evaluate_roi(model_shuffled, X, y)  # 評価は元のyで

    print(f"通常モデルROI: {roi_normal:.1f}%")
    print(f"シャッフルモデルROI: {roi_shuffled:.1f}%")
    print(f"差: {roi_normal - roi_shuffled:.1f}ポイント")

    if roi_normal - roi_shuffled < 5.0:
        print("WARNING: 差が小さすぎます。データリークを強く疑ってください。")
    else:
        print("OK: 正常な差があります。リークの可能性は低いです。")

    return roi_normal, roi_shuffled
```

シャッフルテストの原理はシンプルだ。ターゲット（1着か否か）をランダムにシャッフルすると、「答えに直結する情報」は全部バラバラになる。それでもモデルの精度が高いなら、特徴量の中に「答えそのもの」が含まれている証拠だ。

私のv1コードにシャッフルテストを適用した結果：
- 通常モデル: ROI 148.3%
- シャッフルモデル: ROI 143.1%

**差が5.2ポイントしかない。** これは明確にデータリークのサインだった。

```python
def check_feature_importance(model: lgb.Booster, feature_cols: list[str]):
    """
    特徴量重要度を確認する。
    「place」「payout」「result」など、答えに直結する変数が上位なら即アウト。
    """
    importance = pd.DataFrame({
        "feature": feature_cols,
        "importance": model.feature_importance(importance_type="gain"),
    }).sort_values("importance", ascending=False)

    print("=== 特徴量重要度 TOP20 ===")
    print(importance.head(20).to_string(index=False))

    # 危険な特徴量名を検出
    suspicious = ["place", "payout", "result", "winner", "rank", "is_winner"]
    for feat in importance["feature"].values[:5]:

```

v1のときは `racer_win_rate_30d` が断トツ1位の重要度だった。これ自体は正常に見えるが、「なぜこんなに重要度が高いのか」をよく考えると——当日の勝率が当日の予測特徴量に入っていれば、重要度が高くなるのは当たり前だ。

## 今後の改善：Value Betという考え方

ROIが低い根本原因は「単勝命中率82%は高いが平均配当が581円→BreakEvenの610円に29円足りない」という構造問題だ。

つまり**高確率で当てているが、オッズが低すぎて利益にならない**。AIが「この艇は強い」と判断すると、市場も同じ判断をして低オッズになる。

これを解決するのが **Value Bet（期待値ベースの賭け方）**だ：

```python
def is_value_bet(model_prob: float, win_odds: float, ev_threshold: float = 1.20) -> bool:
    """
    期待値（EV）が閾値以上のときのみ賭ける。
    EV = モデルの予測確率 × 実際のオッズ

    例: モデル確率40%、オッズ3.5倍 → EV = 0.40 × 3.5 = 1.40（閾値1.20超→賭ける）
    例: モデル確率50%、オッズ1.8倍 → EV = 0.50 × 1.8 = 0.90（閾値未満→賭けない）
    """
    ev = model_prob * win_odds
    return ev >= ev_threshold
```

「確率が高い=賭けるべき」ではなく「確率が高い かつ オッズが割高=賭けるべき」という考え方だ。

これを実装するには締切直前のオッズをリアルタイムで取得する仕組みが必要で、現在構築中だ。
